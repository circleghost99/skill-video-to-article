## 1. Smart Probe 移除 [FFmpeg]（05-01）

**問題**：`extract_assets.sh` 的 Smart Probe（±2s 取 5 幀選 JPEG 最大）在某些影片選到 talking head 而非投影片
**根因**：人物畫面（紅色背景+複雜紋理）JPEG 大小 > 投影片（淺底+文字），啟發式失效
**修復**：移除 Smart Probe，直接用 Gemini 回傳的精確時間戳
**檔案**：`scripts/extract_assets.sh`、`SKILL.md` Step 03
**教訓**：Gemini 已看過影片，不需要額外的啟發式選幀

## 7. GIF start_time 精度 [Gemini]（05-01）

**問題**：GIF 前 3-8 秒是黑屏/talking head，內容尚未出現
**根因**：Gemini 把 start_time 設在「場景切換」而非「動畫內容首個視覺元素出現」
**修復**：
- Gemini prompt 規則 4：❌/✅ 對比 + 精確定義
- `extract_assets.sh` Smart Start-Trim：首幀 JPEG < 12KB 自動推進 1s（最多 3 次）
**驗證**：跨 2 部影片 448 幀驗證 12KB 門檻零誤判

| 類型 | JPEG 大小 | 判定 |
|------|----------|------|
| 黑屏/過渡 | 5.6-10.8 KB | ✅ 被跳過 |
| 最暗內容幀 | 13.4-17.2 KB | ✅ 保留 |
| 一般投影片 | 27-179 KB | ✅ 安全 |

**教訓**：二次防禦（prompt 改善 + 腳本 fallback）比只靠一層更可靠

## 8. GIF 12s 硬限制 [Gemini][FFmpeg]（05-02）

**問題**：Gemini 忽視「5-12s 理想」建議，R3 出現 41s 的打字 GIF
**修復**：三層防禦
1. 提示詞：從「理想」改為「❌ 絕對不可超過 12 秒」
2. Python 後處理：`end_time` 自動截斷為 `start_time + 12`
3. Shell 腳本：`-t` cap 從 15s → 12s
**驗證**：R1 max 14s → R4 max 12s，超標 GIF 從 2/4 → 0/3

## 11. Notion 相對圖片路徑 bug [Notion]（05-02）

**問題**：Agent 推 Notion 時 13 張圖全部「跳過不存在」，被迫手動上傳 Cloudinary
**根因**：文章引用 `frame_02.jpg`（相對路徑），腳本從 cwd 做 `os.path.exists()` → False
**修復**：`notion_hamster_push.py` 新增 `_resolve_image_path()`，基於 `--file` 目錄搜索：
- `article_dir/filename`
- `article_dir/images/filename`
- `article_dir/images/images/filename`（v2a 雙層結構）
**教訓**：路徑解析永遠要基於上下文目錄，不要依賴 cwd

## 13. Notion `--update` 模式 [Notion]（05-02）

**問題**：修復模糊圖片時，Agent 手寫 20+ 步 raw Notion API（block 搜索、PATCH、刪除、重建），還改錯位置
**修復**：`notion_hamster_push.py` 新增 `--update PAGE_ID` 模式：
1. 更新頁面屬性（封面、標籤）
2. 刪除所有舊 blocks
3. 重新 append 新 blocks
**效果**：修圖從 28 步 → 3 步（改文章 → `--update` 重推 → 完成）

## 14. Notion Manifest [Notion]（05-02）

**問題**：Agent 在 50+ blocks 中找圖片位置困難，靠 URL 比對常出錯
**修復**：推送成功後自動生成 `notion_manifest.json`：
```json
{
  "page_id": "...",
  "images": [
    {"index": 1, "block_id": "abc123", "url": "https://...", "caption": "..."}
  ]
}
```
**效果**：需要替換單張圖片時，直接用 block_id PATCH，不需要搜索
