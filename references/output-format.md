# Article-Grade Output Reference

這份文件定義 `video-to-article` v1 的輸出標準。

目標不是摘要，也不是逐字稿改寫。
目標是：**把單支影片重組成一篇可讀、可轉發、可保存的解讀型文章。**

## 1. 什麼叫 Article-Grade (Anti-Overcompression)

目標不是摘要，也不是逐字稿改寫，更不是「去脈絡化」的結論總結。
目標是：**把單支影片重組成一篇具備內容厚度、可操作邏輯、且保留關鍵前提的解讀型文章。**

### 拒絕「過度壓縮」(Anti-Overcompression)
LLM 容易為了「簡潔」而犧牲細節。在撰寫時必須對抗以下傾向：
1. **拒絕去脈絡化**：不能只寫「系統會自動分類」，必須寫出「系統會將輸入分為 task, decision, content candidate 等類型」。
2. **保留機制鏈條**：如果影片講了 A -> B -> C 的流程，文章不能直接跳到 C。必須保留中間的轉化邏輯。
3. **保留邊界與前提**：技術限制（如 Linux 不支援）、使用前提、具體數字（如 4 個選項、5 個步驟）必須保留。

## 2. 必須包含的 5 層資訊 (Context Checklist)

每篇文章在交付前，必須確認是否包含以下層次：
- **A. 問題背景**：為什麼作者要拍這支片？原本的痛點是什麼？
- **B. 輸入與入口**：資訊是如何進入系統的？（具體操作、媒介、入口）
- **C. 轉化機制**：資訊是如何被處理、分流、提煉的？（關鍵的邏輯步驟或技術路徑）
- **D. 關鍵案例與細節**：保留至少 1-2 個具體例子，以及關鍵的數字、選項或參數。
- **E. 限制與未來**：這套方法在什麼情況下無效？目前有什麼限制？

## 3. v1 預設輸出類型：解讀型文章

### Rich Article Skeleton (厚度骨架)

即使是短影片，也應維持具備厚度的結構：

```markdown
---
title: [價值導向標題]
source: ...
duration: ...
tags: [...]
quality_badge: ...
---

# [標題]

## 前言
- **作者背景**：[誰做的這套系統/誰拍的這部影片]
- **影片核心**：[這部影片在講什麼，請用一句話精煉]
- **讀者獲益**：
  - [學到什麼 1]
  - [學到什麼 2]
  - [學到什麼 3]

## 核心命題：解決什麼「知識焦慮」或「效率瓶頸」？
[描述背景與痛點，點出這支影片的核心主張]

## 系統入口：資訊如何被捕獲？
[描述具體的輸入方式，例如：對話式、拖拽式、自動抓取等]
[案例：例如影片中提到的河馬桌寵或是特定的網頁抓取案例]

## 關鍵機制：從原材料到長期能力的轉化路徑
[這是文章最厚的部分。不能只有標題，要寫出層次感]
[例如：Intake (初步分類) -> Distillation (精煉) -> Long-term Memory (存儲)]
[必須寫出：處理的具體選項（如：合併、升級）]

## 實際價值：為什麼這套方法比傳統方式更好？
[分析機制的優點，例如：從「看過」變「會做」，從「筆記」變「技能」]

## 使用限制與實作建議
[保留技術限制（如系統平台）、操作前提、或是作者給出的客製化建議]

## 結論與洞察
[總結並給出一個超越影片內容的深度見解]
```

## 4. H2 寫作與診斷規則

### 每個 H2 的「必答三問」
為了確保段落不「太扁」，每個 H2 下下的內容應回答：
1. **機制是什麼？** (Mechanism) - 它是怎麼跑的？
2. **具體細節為何？** (Specifics) - 有哪些選項、數字、案例？
3. **為什麼重要？** (Why) - 這一步解決了什麼問題？

### 「太扁」的診斷訊號 (Warning Signs)
如果出現以下情況，說明文章已被過度壓縮，需要補回細節：
- H2 下方全是 Bullet points，沒有自然段落的推論。
- 只有「功能清單」，沒有「轉化邏輯」。
- 刪除了所有的技術名稱、限制說明或具體數字。

不是每篇都要完全照抄這個標題，但邏輯順序應該接近。

## 3. Title 與 Intro 規則

### Title
標題要是**價值導向**，不是影片名照貼。

#### 好的方向
- 為什麼 X 會變成主流
- X 真正解決的是什麼問題
- 從這支影片看懂 X 背後的設計邏輯

#### 不好的方向
- 關於某某影片的摘要
- 某某人在影片裡講了什麼
- 精彩解析某某主題

### Intro
開頭用 2-3 段完成三件事：
1. 點出影片處理的問題
2. 點出作者主張
3. 點出這件事為什麼值得讀者關心

不要用這些空話開頭：
- 在這支精彩的影片中
- 深入探討
- 全面解析
- comprehensive overview

## 4. H2 寫作規則

### 每個 H2 都要有功能
不是把字幕切段而已。

每個主段應回答其中一種問題：
- 問題是什麼？
- 作者怎麼看？
- 為什麼這樣成立？
- 對讀者有什麼實際影響？

### 段落要自然，不要 bullet 化
- 每個 H2 下面至少 2-4 個自然段
- 只有在真的需要比較或條件拆分時才用 bullets

### 避免「作者說」流水帳
少寫：
- 作者先說 A，再說 B，接著說 C

改寫成：
- 為了處理 A，影片提出 B；而 B 真正有意思的地方在於 C

## 5. 短影片特例（< 8 分鐘）

對 5-8 分鐘的短影片：
- 不要硬撐太多 H2
- 通常 **3 個主段 + 1 個結論** 就夠
- 文章應短而密，不要水

建議骨架：
1. 這支片的核心問題
2. 作者提出的關鍵機制 / 架構
3. 這件事為什麼重要
4. 結論

## 6. 洞察寫法

每篇至少要有一個明確的「關鍵洞察」。

可用這種句型：
- 真正重要的不是 X，而是 X 背後把 Y 變成了可重複流程。
- 這支影片表面上在講 A，但真正有價值的是它重新定義了 B。
- 與其說這是一個工具展示，不如說它在示範一種新的工作方式。

## 7. OpenClaw / 技術影片的專用規則

如果影片是工具、工作流、AI 系統解說：
- 保留產品名、功能名、專有名詞原文
- 中文解釋後可保留英文括號
- 重點放在：
  - 解決什麼問題
  - 入口是什麼
  - 流程怎麼運作
  - 為什麼這樣設計
  - 最終會養成什麼能力

## 8. YAML Frontmatter

至少包含：
- `title`
- `source`
- `duration`
- `tags`
- `quality_badge`

## 9. Review Checklist

交付前自查：
1. 沒看影片的人能不能看懂？
2. 這篇像文章，還是像整理過的字幕？
3. 有沒有把真正的核心主張寫出來？
4. 有沒有留下訂閱、寒暄、口語贅字？
5. 如果影片很短，我有沒有硬灌水？

## 10. 格式禁止項（強制）

下游腳本 `notion_hamster_push.py` 會解析 Markdown 並轉換為 Notion block。
以下格式會導致解析錯誤，**文章中嚴禁使用**：

| 禁止項 | 原因 | 替代方案 |
|--------|------|----------|
| `---`（三短線分隔線） | 被 Notion 解析器轉成 divider block，破壞版面 | 用 `## 小節標題` 或空行分隔章節 |
| HTML tag（`<sup>`, `<a>`, `<span>` 等） | Notion API 不支援 HTML inline | 用 Markdown 語法替代 |
| 英文破折號 `—`（U+2014） | 與中文排版不一致 | 腳本會自動轉成「，」 |
| 本地 `file:///` 圖片路徑 | Notion 無法存取本地檔案 | 先上傳圖床或使用 Notion 原生圖片 block |

> **注意**：frontmatter 的 `---` 不受此限制，腳本會自動剝離。

## 圖片素材規範（強制）

### 存放位置
- **嚴禁**放在 `references/` 資料夾
- **必須**放在 `outputs/images/`（與文章輸出在同一目錄）
- 理由：`references/` 是技能邏輯資料，`outputs/images/` 是文章附屬素材

### 路徑格式
- **必須**使用絕對路徑：`file:///Users/circleghost/Desktop/開發/SKILL/tina/video-to-article/outputs/images/圖片名.jpg`
- **禁止**使用相對路徑（如 `../images/xx.jpg` 或 `references/anchors/xx.jpg`）
- 理由：相對路徑在文章被複製、轉發、或發送到 Discord 時會斷裂

### 命名規則
- 格式：`outputs/images/{影片ID}_{時間戳}_{描述}.{副檔名}`
- 範例：`N519Nj7LRXA_151s_skill-pack.jpg`

### 絕對路徑模板
```
file:///Users/circleghost/Desktop/開發/SKILL/tina/video-to-article/outputs/images/{檔名}
```
